Executive Summary
This Software Design Description (SDD) document presents the architectural and design-level specification of the ObservAI system, an AI-based customer analytics platform developed to support data-driven decision making in cafés and restaurants. The document describes how the system is structured, how its major components interact, and how the selected requirements are realized through software design models. The SDD covers the overall software architecture using a layered block diagram, detailing the responsibilities of the frontend dashboard, backend services, database, and AI processing components. It also includes a deployment view illustrating how the software components are physically distributed across client devices, application servers, databases, and external camera sources. In addition, the document provides analysis-level UML models—activity diagrams, system sequence diagrams, and class diagrams—focused on the use cases implemented in the first increment, namely manager authentication, real-time operations dashboard visualization, and zone labeling. The database design is specified through an Entity-Relationship model and relational schema, defining entities, attributes, primary keys, foreign keys, and integrity constraints required to support authentication and analytics visualization. Furthermore, the document explains how design decisions align with non-functional concerns such as security, data integrity, performance, and extensibility, ensuring that the system can evolve without major architectural changes. Design assumptions and constraints are explicitly stated to clarify system limitations and implementation expectations. Traceability between architectural elements and functional requirements is maintained through the document. Error handling and data validation are incorporated at the design level to reduce runtime failures. Through these design artifacts, the SDD demonstrates that the system requirements defined in the Software Requirements Specification are traceable to concrete design decisions and that the selected architecture supports modularity, scalability, and maintainability. The findings of this design effort indicate that the proposed architecture and data model are sufficient to implement the first increment functionality while providing a foundation for future extensions such as advanced analytics, historical reporting, and AI-driven insights in increments.

High-Level Design (Architecture)

Figure 1 Use Case Diagram
Figure 1 presents the overall Use Case Diagram of the ObservAI system with increment-based color coding consistent with SPMP v2. The diagram illustrates how different actors interact with the system and how the functionalities are gradually delivered across increments.
In the 1st Increment Prototype, the core foundation of the system is implemented. First, UC-01 Authenticate Manager enables managers to securely log in to the system and access authorized functionalities. After successful authentication, UC-02 View Operations Dashboard provides a basic dashboard structure where managers can view initial operational information and system placeholders. In addition, UC-08 Zone Labeling allows managers to define and label entrance, exit, and activity zones, which is a critical prerequisite for later analytics and alert mechanisms. These use cases together form a demonstrable and functional baseline for the system.
UC-03 Switch Visualization Mode enhances dashboard usability by allowing different visualization types. UC-04 Receive Operational Alerts and UC-07 Generate Alert support real-time monitoring by notifying managers about abnormal conditions. UC-05 View AI Insights provides AI-generated recommendations and summaries, while UC-06 Analyze Customer Activity represents the core analytics functionality based on camera input and zone data. These advanced use cases build upon the capabilities introduced in the first increment and progressively extend the system toward full functionality.
 Selected Architecture
For ObservAI, we selected a Layered (3-tier) Web Architecture augmented with a separate AI Processing component. Concretely, the system is organized as:
Presentation Layer (Frontend): React-based web dashboard (UI + visualization + interaction).
Application Layer (Backend API): Node.js/Express service exposing HTTPS/JSON REST endpoints for authentication, configuration (sites/cameras/zones), dashboard data retrieval, and alert/insight delivery.
Data Layer (Database): PostgreSQL (with TimescaleDB for time-series analytics data) and a lightweight local cache layer (SQLite) where applicable.
AI Processing Component (Computer Vision & Analytics): A Python-based module (YOLOv8/OpenCV/PyTorch) responsible for consuming camera streams (RTSP), generating anonymized/aggregated analytics outputs (e.g., counts, distributions, heatmaps), and persisting results through the backend/data layer interface.
This architecture intentionally keeps the AI pipeline isolated from the web/API concerns, while still supporting a coherent end-to-end flow from camera stream → analytics generation → storage → dashboard visualization.
Justification Based on Non-Functional Requirements
Maintainability (Layered Design / Modularity)
A layered structure enforces separation of concerns: UI changes remain in the frontend, business rules and security remain in the backend API, and data persistence remains in the database layer.
The AI Processing component is logically separated, allowing its models and preprocessing logic to evolve without forcing disruptive changes in the dashboard code.
Portability
Using web technologies for the presentation layer keeps the client portable across devices (desktop and mobile browsers).
The backend and AI processing can be deployed independently (e.g., on a single server for early increments, or on separate nodes later) without changing the overall design.
Learnability, Feedback, and Consistency
Centralizing authentication, authorization, and data access in the backend API reduces duplicated logic and enables consistent behavior across UI screens.
Clear API contracts enable predictable and consistent UI feedback patterns (errors, loading states, notifications), which supports learnability.
Localization (TR/EN)
Localization is naturally handled in the presentation layer (and, when needed, supported by backend-delivered message keys), keeping language concerns out of the AI and database layers.
Privacy & Compliance (KVKK/GDPR-oriented constraints)
The separation of AI Processing helps enforce a strict boundary where only anonymized/derived analytics are propagated forward.
The backend/data layer acts as a controlled gate for what is persisted and exposed to the dashboard, supporting “privacy-by-design” decisions (e.g., not storing identifying visuals).
Advantages of the Selected Architecture
Clear separation of responsibilities: UI, API/business logic, AI analytics, and data storage can be developed and tested in parallel.
Better change isolation: Updating AI models or improving analytics does not require redesigning the frontend; UI changes do not endanger AI correctness.
Scalable evolution path: Early increments can run as a tightly integrated deployment, while later increments can scale AI processing and analytics storage independently.
Improved testability: Layer boundaries support unit testing (frontend components, backend services, AI modules) and structured integration/system testing through stable interfaces.
Disadvantages / Trade-offs
Higher integration effort: Separating AI processing introduces additional integration points and requires careful interface definition (data formats, latency expectations, failure handling).
Operational complexity: Multiple components can increase deployment and monitoring complexity compared to a single monolithic application.
Real-time expectations vs. stream variability: Camera stream instability (RTSP interruptions, lighting/angle changes) can affect the freshness and reliability of analytics updates, requiring defensive design and clear user-facing feedback.
Impact on the Development Process
We will follow a contract-first mindset for the interfaces between layers (especially frontend ↔ backend ↔ AI processing), so different sub-teams can progress without blocking.
Increment development will prioritize a stable backend API and database schema for the 1st Increment use cases (Authentication, Dashboard Viewing, Zone Labeling), while keeping AI-related interfaces extensible for future analytics features.
The architecture supports incremental refinement: early increments validate end-to-end data flow and dashboard usability; later increments deepen AI capability and insight generation without restructuring the entire system.
 Logical View

Figure 2 Block Diagram
The software architecture of ObservAI follows a modular, layered design that separates concerns between presentation, application logic, data management, and AI processing.
The Manager Panel provides the primary interaction point for café or restaurant managers. Through this interface, users authenticate themselves, configure zones, monitor analytics, and view AI-generated insights.
The Web Dashboard acts as the presentation layer and communicates with the backend using secure RESTful APIs. It displays real-time metrics, charts, heatmaps, and alerts based on processed analytics data.
The Backend API Server is responsible for business logic, request handling, authentication, authorization, and orchestration of data flow. It processes incoming requests from the dashboard, queries the database, and coordinates with the AI Analytics Engine.
The Database System stores persistent data including users, cameras, zones, analytics records, and heatmap data. It ensures data integrity and supports historical analytics queries.
The AI Analytics Engine performs computer vision and AI-based processing. It consumes live video streams from IP Cameras, runs detection and tracking algorithms, and generates structured analytics outputs. These results are sent back to the Backend API Server for storage and visualization.
This architecture enables scalability, maintainability, and clear separation of responsibilities, ensuring that each component can evolve independently while supporting real-time analytics and decision support.


Figure 3 EER Diagram
The Enhanced Entity-Relationship (EER) diagram models the core data structures of the ObservAI system and the relationships between them by using Chen’s notation. The main purpose of the database is to support user authentication, camera-based analytics, zone definitions, and AI-generated insights.
The User entity represents managers who access the ObservAI dashboard. Each user is identified by a unique user_id and has attributes such as email, password_hash, created_at, and last_login. A user can view multiple Analytics_Records, establishing a one-to-many (1:M) relationship between User and Analytics_Record. Participation of User in this relationship is partial, since analytics records may exist even if they are not actively viewed.
The Camera entity represents IP cameras installed in the monitored environment. Each camera is uniquely identified by camera_id and includes attributes such as location_name, rtsp_url, and status. A camera produces multiple Analytics_Records, forming a one-to-many relationship. This relationship has total participation on the Analytics_Record side, as each analytics record must originate from a camera.
The Zone entity models logical regions such as entrances, exits, or activity areas defined by the manager. Each zone has a unique zone_id, along with attributes like zone_name, coordinates, and camera_id. A camera may define multiple zones, while each zone is associated with exactly one camera, resulting in a one-to-many (1:M) relationship.
The Analytics_Record entity stores processed analytics outputs generated by the AI pipeline. It includes attributes such as record_id, timestamp, people_count, avg_age_range, and gender_distribution. Each analytics record is associated with exactly one camera and one zone, ensuring traceability of analytical results.
The Heatmap_Data entity represents aggregated density information over time windows. Attributes such as heatmap_id, time_window, density_matrix, and camera_id capture spatial crowd distributions. Each camera can generate multiple heatmaps, while each heatmap belongs to exactly one camera.
Overall, the EER diagram ensures data consistency by enforcing appropriate cardinalities, participation constraints, and attribute definitions, while supporting scalable analytics, zone-based monitoring, and AI-driven insights within ObservAI.
 Physical View

Figure 4 Deployment Diagram
The UML Deployment Diagram illustrates the physical deployment of the ObservAI system components and their communication paths at runtime.
The Manager Device represents end-user devices such as laptops or tablets used by managers. These devices access the system via a web browser and communicate with the Web Application Server over secure HTTPS connections.
The Web Application Server hosts the frontend application and exposes RESTful APIs. It acts as the central hub for request handling, forwarding analytics queries to the backend services and persisting data to the database.
The Database Server stores all persistent system data and communicates with the Web Application Server using SQL-based connections. This separation improves data security and performance.
The AI Processing Server is responsible for executing computationally intensive AI workloads. It receives real-time video streams from Camera Devices using RTSP and processes them independently from the web server to ensure system responsiveness.
Camera Devices continuously stream video data to the AI Processing Server, which extracts analytics and forwards results to the Web Application Server. This deployment structure ensures efficient real-time processing, fault isolation, and scalability of AI components.

NFReq1 – Ease of Learning
Design Approach: The UI is designed as a manager-only dashboard with a consistent navigation structure and minimal steps for common tasks (login → dashboard → visualization switch → export). We use familiar dashboard patterns (clear charts, recognizable icons, concise labels) and provide lightweight onboarding/help cues where needed.
Verification & Validation: Conduct onboarding usability tests with first-time managerial users and measure time-to-complete for core tasks (log in, navigate dashboard, switch visualization modes, export report). Collect qualitative feedback on the clarity of icons/labels and whether the user can operate the system without external documentation.
NFReq2 – Consistency and Feedback
Design Approach: We enforce a shared layout and reusable UI components across all pages (same structure, spacing, and interaction patterns). A unified feedback mechanism (loading states, success/error toasts, progress indicators, status badges) ensures users always see immediate and predictable responses to actions (e.g., camera status, analytics refresh, report generation).
Verification & Validation: Perform a full UI walkthrough using a consistency checklist (layout, buttons, dialogs, error handling). Trigger actions that require instant feedback (camera connect/disconnect, refresh, export) and confirm the UI responds immediately and consistently. Use browser dev tools/performance checks to ensure feedback is not delayed.
NFReq3 – Maintainability
Design Approach: The system follows a layered structure (Presentation: React, Logic: Node.js/Express API, Data: PostgreSQL/TimescaleDB) with clear module boundaries and stable interfaces. AI processing is isolated as a separate component so model/analytics changes do not force large refactors in the UI or API. We also rely on a branch-based workflow with PR reviews to keep code quality consistent.
Verification & Validation: Inspect the repository structure to confirm separation of concerns. Perform a controlled change (e.g., replace/update a dashboard component or an API endpoint) and verify unrelated modules remain unaffected. Use static analysis and quality gates (linting, type checks if applicable) plus PR review records to validate maintainability in practice.
NFReq4 – Portability
Design Approach: The frontend is browser-based and the backend exposes standard HTTPS/JSON APIs; configuration is environment-based (no OS-specific assumptions). This supports deployment across different environments (development machines and servers) without code changes.
Verification & Validation: Build and deploy the frontend/backend on at least two environments (e.g., Windows and macOS/Linux server setups) and run the same smoke tests/API calls to confirm identical behavior. Verify that installation/build steps do not require OS-specific code modifications.
NFReq5 – Localization
Design Approach: All user-facing text is handled through localization keys and translation files (TR/EN). Charts, alerts, and dashboard labels are generated using localized strings to avoid mixed-language UI. User language preference is supported at the UI level (and can be persisted if needed).
Verification & Validation: Switch the application language between Turkish and English and verify that all screens update correctly (menus, dialogs, alerts, chart labels). Check for missing translations and validate locale-dependent formatting (dates/numbers) where applicable.
2. Test Case Design
This section describes the approaches that will be used to design test cases for ObservAI, as well as the formatting rules that will be followed when documenting these test cases. Unit testing, software integration testing, system testing, and acceptance testing are all included in the testing strategy, which is in line with the incremental development model.
2.1. Unit Tests
Approach:
Unit tests will be designed to verify the correctness of individual software components in isolation. Unit testing in ObservAI will primarily concentrate on data-handling logic, AI processing modules, and backend services. Using predetermined inputs and expected outputs, each unit test will verify a single function or class against its anticipated behavior.
The following techniques will be used:
White-box testing based on source code logic
Equivalence partitioning for input validation functions
Boundary value analysis for threshold-based logic
External dependencies such as camera streams, databases, and APIs (Gemini LLM API, notification services) will be mocked to ensure isolation.
Formatting Rules:
Each unit test case will be documented using the following structure:
Test ID
Test Objective
Input Data
Expected Output
Actual Output
Test Result (Pass/Fail)
2.2. Software Integration Tests
Approach:
After unit-level validation, software integration tests will be created to confirm how integrated components interact. Integration testing in ObservAI will concentrate on:
Communication between the frontend dashboard and backend services
The proper transmission of AI inference results to analytics and visualization components
Camera input integration with the AI pipeline
Interaction with external services like LLM-based insight generation and notification
A bottom-up integration strategy will be used, with frontend integration coming after backend and AI components.
Formatting Rules:
Each integration test case will include:
Test Case ID
Integrated Components
Preconditions
Expected System Behavior
Actual Result
Test Status (Pass/Fail)
2.3. System Tests
Approach:
System testing will validate that the entire ObservAI system satisfies the requirements outlined in SRS A v2 and SRS B v2. Black-box testing will be used in the design of these tests, with an emphasis on system-level functionality rather than internal implementation specifics.
System test cases will be derived directly from:
Functional Requirements (FReq IDs)
Use Cases (UC-01 to UC-08)
Performance under concurrent usage and accuracy of analytics outputs will be tested in both normal operation scenarios and error conditions.
Formatting Rules:
Each system test case will follow this format:
Test case ID (e.g., ST-01)
Related Requirement / Use Case
Test Scenario Description
Expected Result
Actual Result
Test Outcome (Pass/Fail)
2.4. Acceptance Tests
Approach:
From the standpoint of the café/restaurant manager, acceptance tests will confirm whether ObservAI meets user expectations and project goals. User stories and acceptance criteria established during Agile estimation and requirements analysis will serve as the foundation for these tests.
Acceptance testing will simulate real-world usage scenarios, such as:
Logging into the system
Monitoring live visitor analytics
Receiving alerts for overcrowding
Interpreting AI-generated insights for decision-making
Acceptance tests will be carried out during final system validation as well as at the conclusion of each major increment.
Formatting Rules:
Acceptance test cases will be documented as follows:
Test Case ID (e.g., AT-01)
User Story / Use Case Reference
Acceptance Criteria
Test Steps
Expected Outcome
Observed Outcome
Acceptance Status (Accepted / Not Accepted)
Summary:
All test cases will follow a consistent and structured format to ensure traceability, clarity, and reproducibility. Test case identifiers will clearly indicate the test level (UT, IT, ST, AT), and each test case will be traceable to related requirements or use cases. This approach ensures systematic validation of ObservAI at component, integration, system, and user acceptance levels.

3.2. Process View
UC-01: User Login & Authentication

Figure 8 UC-01 Communication Diagram
This use case is realized through the interaction between the Manager actor, the LoginUI boundary object, the AuthController control object, and the UserRepository entity. The Manager initiates the process by entering login credentials via the LoginUI. The LoginUI forwards the credentials to the AuthController, which is responsible for validating authentication logic. The AuthController queries the UserRepository to verify the credentials. Based on the validation result, the AuthController informs the LoginUI, which then displays the authentication result to the Manager. This realization separates user interaction, control logic, and data access responsibilities.

UC-02: Real-Time Analytics Dashboard Flow

Figure 9 UC-02 Communication Diagram
This use case is realized by the interaction between the Manager, DashboardUI, DashboardController, and AnalyticsRepository. The Manager requests to view the operational dashboard through the DashboardUI. The DashboardController retrieves real-time analytics data from the AnalyticsRepository and processes it for visualization. The processed data is then sent back to the DashboardUI, which presents live metrics and visual analytics to the Manager. This realization supports real-time monitoring functionality within the first increment.

UC-08: Zone Labeling & Configuration Flow

Figure 10 UC-08 Communication Diagram
This use case is realized through the collaboration of the Manager actor, ZoneLabelUI boundary, ZoneController control, and ZoneRepository entity. The Manager defines entrance and exit zones using the ZoneLabelUI. The ZoneController processes and validates the zone information and persists it via the ZoneRepository. Upon successful storage, confirmation is returned and the Manager is informed. This functionality enables accurate zone-based analytics in later increments.

5. Discussions
Document Consistency & Traceability
Maintaining alignment across the Initial Plan, SRS-A, SRS-B (Prototype), and SPMP was a key challenge.
We repeatedly cross-checked scope statements, terminology, and FReq/NFReq mappings so that design decisions in the SDD reflect the same source of truth.
Increment Scope vs. Full Product Vision
The 1st Increment focuses on a limited use-case set (Authentication, Viewing the Operations Dashboard, Zone Labeling), while the SRS documents describe the broader end-product vision.
Therefore, we explicitly separated “implemented in this increment” vs. “planned for future increments” in the design narrative.
AI/CV Pipeline Variability & Testability
RTSP camera streams are inherently variable (lighting, angle, occlusions, crowd density, connectivity), which can affect the stability of AI-driven outputs.
This required clear component boundaries (frontend–backend–AI processing) and early thinking about measurable verification criteria for system outputs.
Privacy / KVKK–GDPR Constraints
Privacy constraints (anonymous processing and avoiding storage of identifying visuals) directly shaped the system design.
These constraints influenced what data is stored vs. derived, as well as logging, retention assumptions, and how analytics results are represented.
Data Design & Retention Decisions
Decisions about analytics granularity (aggregated/derived records), retention periods, and table relationships needed to balance performance, scalability, and privacy.
The database design was structured to support incremental expansion in later increments (e.g., additional analytics types and insights).
Template-Driven Expectations
The template’s required structure (High-Level vs. Low-Level separation, mandatory sections, and explicit V&V mapping for NFReqs) required strict formatting discipline.
A previously high-scoring SDD was used as a structural reference—especially for the HLD/LLD organization—while adapting content to our project scope and template rules.
Process Observations
Keeping artifacts synchronized required disciplined version control, regular reviews, and consistent meeting/sprint routines.
We also noted that some numerical details (e.g., cost totals) can diverge across documents unless explicitly reconciled in the final version.
